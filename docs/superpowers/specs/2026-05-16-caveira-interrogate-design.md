# Caveira — `interrogate` Subcommand Design Spec

**Date:** 2026-05-16
**Status:** Approved for planning
**Scope:** Add a read-only `cav interrogate` subcommand that scans a repository's
commit history and reports the human identities (with commit counts) and the AI
models present, plus an `--emit-mailmap` mode that prints a `.mailmap` skeleton.

## 1. Purpose

Before fabricating a history, a user needs to know *who* Caveira sees in a repo.
Caveira keys players by email, so a single person whose git `user.email` drifted
across commits is counted as two players. Today the only way to inspect this is
the interactive picker, which appears mid-fabricate and cannot be summoned on
demand. `interrogate` exposes the existing scan as a standalone, read-only
command: list every discovered identity and its commit count, list the AI models
(which are excluded from being players), and optionally emit a `.mailmap`
skeleton to bootstrap unifying drifted identities.

## 2. Architecture

`interrogate` is a **child subcommand** of the existing cobra root command. The
root command keeps its current behavior (`cav --repo … --start … --end …`, the
retime/fabricate path) entirely unchanged — `interrogate` has its own flag set
and never sees `--start`/`--end`.

One new file, `internal/cli/interrogate.go` (+ `interrogate_test.go`), builds the
subcommand, runs the scan, and formats output. It is wired into the root via
`cmd.AddCommand(...)` inside `newRootCmd`.

`interrogate` is strictly **read-only**: it never schedules, rewrites, swaps the
working folder, or pushes. It reuses, unchanged:

- `acquireSource` (`pipeline.go`) — resolves `--repo` to a local path, cloning a
  URL read-only if needed. `interrogate` constructs a minimal
  `input.Config{Repo, OutDir}` to call it.
- `git.PlainOpen` — opens the resolved local path.
- `fabricate.LoadMailmap(srcPath)` — loads `<repo>/.mailmap` (nil if absent).
- `fabricate.DiscoverIdentities(repo, mm)` — the human-identity scan (already
  excludes models, sorts by commit count desc then name asc).
- `fabricate.ScanModelReport(repo, mm)` — its `ModelReport.Models` field is the
  set of AI models present in history.

No changes to `internal/fabricate` or `internal/input` are required; this is
purely additive in `internal/cli`.

## 3. Command surface

```
cav interrogate --repo <path-or-url> [--out-dir <dir>] [--emit-mailmap]
```

- `--repo` — path or URL of the repository to scan. **Required.**
- `--out-dir` — parent directory for URL clones (default `$CWD`), mirroring the
  root command's flag of the same name. Only relevant when `--repo` is a URL.
- `--emit-mailmap` — print a `.mailmap` skeleton instead of the report (§5).

`interrogate` does **not** accept `--start`, `--end`, `--seed`, `--fabricate`,
`--pigs`, or any rewrite flag. It is a separate subcommand with a separate flag
set, so those flags are simply not registered on it.

## 4. Default output

`cav interrogate --repo X` writes a plain-text report to stdout:

```
Identities in <repo-display>  (.mailmap applied)

Players (2):
  justin06lee <hi@justin06lee.dev>            131 commits
  justin06lee <justin.leehuiyun@gmail.com>      1 commit

AI models (excluded from players — co-author only):
  Claude <noreply@anthropic.com>
```

Details:

- **Header line** — names the repo (the resolved local path, `srcPath`) and
  states whether a `.mailmap` was found and applied. When `LoadMailmap` returns
  a nil map (no file), the header reads `(no .mailmap)` instead of
  `(.mailmap applied)`.
- **Players section** — every `DiscoveredIdentity` from `DiscoverIdentities`, in
  its existing sort order (commit count descending, then name ascending). Each
  line: `<name> <<email>>` followed by the count. The count word is pluralized
  (`1 commit` / `N commits`). The section header includes the total count.
- **AI models section** — every identity in `ModelReport.Models`, in that
  field's existing sort order (lowercased email ascending). The header label
  states plainly that models are excluded from being players and appear only as
  co-authors. If no models are present, the section is omitted entirely (no
  empty section).
- **Empty history** — if `DiscoverIdentities` returns no identities, the report
  prints a single friendly line (e.g. `No commits found in <repo>.`) and the
  command exits 0. An empty or commitless repo is a valid answer, not an error.

## 5. `--emit-mailmap` skeleton

`cav interrogate --repo X --emit-mailmap` prints a `.mailmap` skeleton to stdout
**instead of** the §4 report, and prints nothing else — so
`cav interrogate --repo X --emit-mailmap > .mailmap` produces a clean file.

```
# .mailmap skeleton generated by `cav interrogate --emit-mailmap`
# One line per identity discovered in history. Caveira does not guess which
# identities belong to the same person — to unify a drifted identity, merge its
# lines onto the canonical one, e.g.:
#   Proper Name <canonical@email> <other@email>
justin06lee <hi@justin06lee.dev>
justin06lee <justin.leehuiyun@gmail.com>
```

Details:

- One `<name> <<email>>` line per **human** identity from `DiscoverIdentities`,
  in the same sort order as §4. AI models are excluded — they are not players
  and do not belong in a player `.mailmap`.
- A comment header explains the format and that the user must draw the merges
  themselves. Caveira never guesses that two emails are one person — consistent
  with the existing `.mailmap` feature's explicit non-goal.
- If history has no identities, only the comment header is printed (an empty but
  valid skeleton); exit 0.
- `--emit-mailmap` never writes a file. It prints to stdout; redirecting is the
  user's choice. This avoids clobbering an existing `<repo>/.mailmap`.

## 6. Error handling

- Missing `--repo` — cobra's required-flag error; non-zero exit.
- Unresolvable repo (bad path, clone failure, not a git repository) — a clear
  `error:` message to stderr; non-zero exit.
- Empty/commitless history — **not** an error (see §4); exit 0.
- A scan error from `DiscoverIdentities`/`ScanModelReport` — surfaced as an
  `error:` message; non-zero exit.

## 7. Testing

- **`interrogate_test.go` (unit)** — report formatting: players section with
  pluralized counts, the model section label, the `(.mailmap applied)` vs
  `(no .mailmap)` header, model section omitted when no models, the
  empty-history line. The `--emit-mailmap` skeleton: one line per identity, the
  comment header, models excluded.
- **CLI parsing** — `cav interrogate` is registered as a subcommand; `--repo`
  is required; `--emit-mailmap` parses; rewrite flags are absent from its flag
  set.
- **Integration** (`internal/cli/`) — a fixture repo whose history has a drifted
  human identity (two emails) and an AI co-author: `cav interrogate` lists both
  human identities with their counts and the model in its own section;
  `cav interrogate --emit-mailmap` prints one line per human identity and
  excludes the model. Confirm the repo is not modified by the run.

## 8. Non-goals

- No JSON or machine-readable output — plain text only.
- No auto-merging of identities in the skeleton; Caveira never guesses that two
  emails are one person.
- `--emit-mailmap` does not write `<repo>/.mailmap`; it prints to stdout.
- `interrogate` never modifies the repository in any way.
- No per-player model-usage breakdown (`ModelReport.Profiles`) — only the set of
  models is reported. A usage breakdown can be a later enhancement if wanted.
- The root retime/fabricate command and its flags are untouched.
